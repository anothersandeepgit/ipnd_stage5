1.2.1 Creating a Structured Document with HTML
	The input to webpage development is a webpage design as PDF/image file.
	The output is the result of Frontend development which involves 
	HTML, CSS, JavaScript. In this nanodegree we only cover HTML and CSS.
	The browser uses HTML document to build a tree-like structure called 
	DOM (Document Object Model), each node of which is an HTML element.
	Every HTML element in a webpage is rendered in a rectangular area 
	of the webpage.
1.2.2 Languages:
	HTML: language for the "structure" of a webpage.
	CSS: language for the "style" (appearance) of a webpage.
1.2.3 Tools:
	Besides the languages (HTML and CSS), Tools are very important for webpage development.
	Editing tool:
		--scratchpad.io is a good editor to start development since it is easy and 
		feedback is immediate. It can not be used to share the web page though.
		--codepen.io is an intermediate level editing tool. It allows sharing and 
		saving of developed webpage. We need to create an account on its web page though.
		--Sublime Text 2 is the tool of choice for Professional webpage development.
				
	
	Troubleshooting/Debugging tool:
		--Chrome has very powerful "Developer Tool" that when launched from Menu appears 
		as structure and style panels in addition to the panel where the webpage is 
		rendered. The panels allow in-place changing of structure and style to see the 
		effect on the resulting webpage.
		--Developer Tool is actually a Toolbox. One of the tools in the toolbox allows 
		inspection of structures and styles of various elements of the webpage. In the 
		reverse direction hovering over a structural element in HTML panel highlights 
		the box in the result panel.
1.2.4 Development steps:
	Everything in a webpage is a box. Start development from one big box that is the 
	entire webpage, and progressively specify details and positions of smaller boxes; 
	each box should be defined with a "div" tag. The contents of boxes are filled next. 
	To add style give each "div" tag a "class" attribute with informative name.

1.3.1 CSS: General
	-CSS (Cascading Style Sheets)code is used to add style (appearance) to HTML elements 
	on a webpage; "Cascading" means for each HTML element the more specific style rule 
	ultimately applies 
	CSS rules: 
		<selector> determines the elements to which the rules apply such as 
			"element tag": 
			"class (ex. .description)": used for styling "many similar elements 
				of possibly different tags". The use of class helps 
				"avoid repetition" of CSS code. 
			"id (#lesson1)": used to style a "unique" element; although possible to 
				use it like a class it is not recommended. Uniqueness is also 
				used by JavaScript.  
				 
		-each rule is a line of the form property:value 
			<selector> { 
				property: value; 
				... 
			} 
				 
				
	Some useful rules: 
		-"p {line-height: ...}" - controls space between consecutive lines of text
		-"background-image: url(images/myImage.png)" 
				

	-various CSS properties are listed at: 
		http://developer.mozilla.org/en-US/docs/Web/CSS/Reference
1.3.2 CSS: Inheritance
	an element "inherits" style rules from its parents which it can override; Browsers 
	have default styles builtin which can be overridden.
	--descendent elements "inherit text-relatd properties" such as: color, font, 
	letter-spacing, line-height, list-style, txt-align, text-indent, text-transform, 
	visibility, white-space, word-spacing
	--descendent elements "do not inherit box-related properties": background, border, 
	display, float and clear, height and width, margin, min/max height/width, outline, 
	overflow, padding, position, text-decoration, vertical-align, z-index
1.3.3 CSS: The Box Model
	-A "box" is basis of applying CSS rules. Boxes have 4 components (from outermost):
		Margin: to provide space between boxes; it is transparent (no background color) 
		Border: our size of element box with rule "box-sizing: border-box" 
		Padding: clears an area around the content 
		Content: 
				
	-div and span don't have any "default" style applied by the browsers
	-"div" elements are for layout control and group things together; their default width 
	is browser width; with "width: ##%" the box resizes as browser is resized, if specified 
	as number of pixels (##px) then no box resizing happens; "max-width" is used to specify 
	an upward bound on a resizeable box size. 				
	-"span" elements are for decorating (styling) elements inline
	
	div { display: flex; } - this rule allows child elements to be placed side by side if 
	space permits; by default "div" elements occupy entire width of the browser
1.3.4 Web Development Process:
	-First structure a webpage; look for natural boxes from Design document (a pdf or 
	a graphic file), then needed HTML tags, followed by sizing and positioning the boxes; 
	the result is a layout
	-look for repeated styles and semantic elements (choose the right tag based on its 
	purpose on the webpage) 
	-write HTML 
	-apply styles (from biggest to smallest boxes) 
	-Fix issues 
	-all resources of a webpge should be at or below the root folder of webpage/site 
	-Developer Tools helps in refinement by allowing experimentation with immediate 
	visual feedback
	-"* { outline: 1px solid red}" CSS rule puts boxes around elements which can be a 
	visual feedback during development 
	-Verify HTML: http://validator.w3.org/#validate_by_input 
	-verify CSS: http://jigsaw.w3.org/css-validator/#validate_by_input

4.1 Overview of the Networks 
	Definition: Network is a group of entities (humans, computers, etc) that can 
	communicate even when they are all not directly connected. Communication relies
	on message forwarding by intermediate entities.

	Message forwarding may be by flooding the network (sending messages to all nodes)
	or some other algorithm.

	Factors in network implementation: 
		message protocol - how server/client entities interact (ex. HTTP GET/POST)
		message priority (resource use rules ex: internet's best effort policy)
		encoding/decoding of messages
		routing of messages (router rules)


	Network performance parameters:
		Latency (startup time usually in milliseconds)
		Bandwidth (information transfer rate usually in bits/sec or Mbps)

	a website to test network performance: www.cnet.com/internet-speed-test



	HTTP URL: 
		<protocol>://<host>:<port>/<path>?<GET/query parameter=values>#<fragment>
	#fragment is interpreted by browser; not sent to server
	spaces in query replaced by '+'; some other chars replaced with their unicode
	browser generates HTTP request after establishing connection with host:
	GET /<path>??<GET/query parameter=values>  HTTP/1.1(or 1.0)
	followed by headers: 
			Host: <hostname>   (required in 1.1 not in 1.0; specifies a site
						since a server can host multiple websites)
			User-Agent: <chrome, etc> (courtesy for server to implement policies)
			***Users can add their own Headers too.

			
	server sends HTTP Response:
		status line:<protocol/version> <response code>   <response phrase>
				HTTP/1.1  200  OK    
			some status codes: 
					200 -> OK -> document found (2XX means success)
					302 -> Found -> document located elsewhere
					 (3XX means to find doc some other step needed)
					404 -> Not Found ->  (4XX error from client side)
					500 -> Server Error  (5XX means server side error)
		followed by mandatory, optional, and user-added headers:
				Date: 
				Server: <optional but to be avoided to prevent hacking>
				Content-Type: image/jpeg  (to help browser handle content)
				Content-Length: (optional)

4.6 Validation (Server Side) 
	Server side validation is important for two reasons:
		Site Security: 
			User can type HTML special characters as part of their Form inputs. 
			These characters ('<', '>', '"', '&') if entered by users should 
			normally not be interpreted, especially if the Server Side 
			application needs to send this data back to the browser for further 
			user editing or reviewing.. These can be interpreted by the browser 
			and so can mess-up the HTML form in unpredictable ways depending on 
			which tags were entered, where, and how many. 
			
			Even if a regular user plays by the rules, a malicious user can 
			create his own Form with malicious data and submit to our Server 
			Side Application. This is also a case against relying completely 
			on Client Side validation alone, as it can be bypassed by malicious 
			users. In addition regular users can turn-off Client-side validation 
			in their browser configuration.

			An even more destructive effect can be if the user input is used to 
			make queries to a Backend Database. A carefully crafted user input 
			can result in undesirable queries which can generate results for 
			user that he is not authorized to get. This is called "SQL Injection".

		Improved User experience: 
			Especially when users have to enter long string as input, they are 
			likely to make mistake as they continue typing in the string. 
			Validation can enable us to accept an input evven if it not 
			completely correct, if our logic can figure out user intent. 
			For example the first three letters of a complete string specifying 
			a month of year can be sufficient to be accepted as a valid input.


			HTML escaping (replacing with equivalents as follows: 
			'<' -> '&lt;', '>' -> '&gt;', '"' -> '&quot;', '&' -> '&amp;'
			is used to ensure browsers do not interpret these HTML special characters 
			and render their values. 
			A technique to do this on a user provided string called "user_input_string":

			import cgi
			escaped_user_input_string = cgi.escape(user_input_string)

			-Hadoop works with data in terms of keys/values

			-each Reducer works on one set of record at a time; it gets the key & list of 
			all values, processes the vaalues in some way, and writes out the final result.

	Mootools JavaScript Framework
	MooTools (My Object-Oriented Tools)[2] is a lightweight, object-oriented JavaScript framework.
	MooTools emphasizes greater modularity and code reuse through object-oriented practices. 
	MooTools contains a robust Class creation and inheritance system.
	MooTools includes a number of components, but not all need to be loaded for each application.
	Some of the component categories are: 
		Core: A collection of utility functions that all the other components require.
		More: collection of add-ons that extend the Core and provide enhanced functionality.
		Class: The base library for Class object instantiation.
		Natives: A collection of JavaScript Native Object enhancements. The Natives add  
			functionality, compatibility, and new methods that simplify coding.
		Element: Contains a large number of enhancements and compatibility standardization  
			to the HTML Element object.
		Fx: An advanced effects-API to animate page elements.
		Request: Includes XHR interface, Cookie, JSON, and HTML retrieval-specific tools
		Window: Provides a cross-browser interface to client-specific information, such as 
			the dimensions of the window.
